# Autopilot v2 — Orchestrator (WALKING SKELETON)

**This is the active engine.** It runs ONE unit of work (ticket-mode) from cold start → PR-ready, then holds for human merge. It is invoked by the `/run-build` slash command and is profile-driven (`profiles/*.yml`). The v1 engine (`ORCHESTRATOR.md`) still works and is self-contained, but v2 is the path under active development.

v2 differs from v1 in three load-bearing ways: **no per-PR token cap** (coarse per-iteration accounting → a night tally → a runaway backstop); a **three-scope durable state layout** (night / global / per-unit); and **worktree-native** per-unit isolation.

---

## 0. What this skeleton does and does NOT do yet

**DOES:** ticket-mode (a rubric picks ONE unit); cold-start → PR-ready; **night-scoped durable state + per-unit state** (§2); a hand-authored profile via `--profile`; the 8-phase engine E0–E7 (verification protocol, phase bracketing, fix caps, seam tests, rewrite-inventory); explicit model pins on every spawn (hook-enforced); a run-active sentinel that arms the run-scoped gates; an end-of-run audit fired by the Stop hook; an arbitration ledger; **HOLD-for-human merge** on every PR; coarse token accounting with a named source (§2); the night stopping rules; a Linear-first tracker with a 3-state probe + a GitHub fallback + a global debt ledger.

**DOES NOT yet (deferred layers, each a SAFE degrade — never a silent no-op or a fabricated pass):**
- `[LAYER: derive-profile]` — REQUIRES a hand-authored profile; else EXIT with a named message.
- `[LAYER: theme-mode]` — ticket-mode only (one unit per iteration; no multi-ticket themed batches).
- `[LAYER: auto-merge]` — holds EVERY PR for human merge.
- `[LAYER: full-slop-gate]` — ports the seam-test + verification-protocol subset; LOGS the rest as deferred.
- `[LAYER: checkpoint-resume]` — whole-iteration wall-clock HALT (3h), not mid-slice checkpoint. Cross-night resume of a halted unit IS supported (§2 resume scan).

**Worktree-native:** the orchestrator enters a per-unit git worktree at **§N2.5** and runs the whole unit there (sub-agent writes, gates, commit, push, PR); the **main checkout is read-only for the run** (any human WIP there is never touched). This makes background and foreground runs identical, and turns any background-isolation guard from a blocker into an ally (it redirects a stray write into the worktree).

## 1. Launch

`/run-build [--profile <name>]` resolves the profile, `cd`s to `{{profile.target.repo_path}}`, then reads this file and executes the **RUN-BUILD PROMPT** block (§4 onward) verbatim. ## 2. State layout — THREE scopes

Night-scoped rules (token backstop, no-progress, night wall-clock, tracker reconciliation, resume) MUST survive across a night's iterations, so they cannot live under a per-iteration id. Three scopes:

**(A) NIGHT-scoped** — under `~/autopilot/state/<run-id>/`, where `<run-id>` is minted ONCE per night and re-entered by every iteration of that night (§N0):
| File | Purpose |
|---|---|
| `night-log.md` | one line per iteration (tracker state + debt count) |
| `loop-state.json` | `{night_start, tokens_used_total, no_progress_count, night_backstop:10000000, reviewer_routable}` — the durable night tally + no-progress counter + the once-per-night deep-reviewer-routability flag (set at N0 step 9; field absent until probed) |
| `theme-ledger.md` | small cross-iteration memory (≤150 lines) |

**(B) GLOBAL** — fixed paths, outlive any night:
| File | Purpose |
|---|---|
| `~/autopilot/state/tracker-debt.yml` | the tracker-outage debt ledger — debt can outlive a night, so it is global, never run-id-scoped |
| `~/autopilot/state/current-night.json` | `{run_id, night_start}` — the night pointer N0 reads to decide adopt-vs-mint |
| `~/autopilot/state/run-active.json` | the run-active sentinel (session-scoped, PID+TTL guarded) |
| `~/autopilot/state/run-worktrees.json` | the isolation-sweep manifest the end-of-run audit reads (written at §N2.5). Overwritten per unit — safe because the N0.3 mutex forbids concurrent runs |

**(C) PER-UNIT** — under `~/autopilot/state/<run-id>/<unit-id>/`, where a **unit is 1..N tickets** (§N3): `<unit-id>` = the ticket-id for a solo unit (unchanged), or `<prefix>-<id1>-<id2>[-…]` for a themed bundle. `unit = branch = PR = state dir = ledger`; every per-unit file below keys off `<unit-id>`, and where any per-unit path elsewhere in this doc says `<ticket-id>`, read it as `<unit-id>` (identical in the common solo case):
| File | Purpose |
|---|---|
| `unit.yml` | the bundle manifest — present for EVERY unit (trivial for a solo): `{members:[ticket-ids], primary, rationale, directed_by, bundle_signal}`. `members` drives the E1/E7 per-ticket loops; `unit_id` = the member ids joined by `-` after the prefix (per §N2's formula) |
| `progress.yml` | append-only phase bracketing + resume — bracketing rule (`start` row FIRST, `complete\|halt\|exit` row LAST, NO exceptions, ADOPTED phases included): §7 opening |
| `spec.md` | the binding spec (E2) |
| `arbitration-<ticket-id>.yml` | one disposition row per review finding (keyed by ticket-id, not PR# which doesn't exist pre-E6) |
| `blocked.md` | hard-stop-needed-mid-flight record (§8 rule 8) |

`[LAYER: theme-mode]` `slices.yml` / `[LAYER: derive-profile]` `profile.derived.yml` are night-scoped when built. `.autopilot/` appearing in the target repo = HALT-EXIT (mis-config).

**Resume scan:** at N0, after adopting/minting the night, scan `state/<run-id>/*/progress.yml` for a unit whose last phase row is `halt` with a branch that still exists and has no open PR → resume it (skip to its next uncompleted phase) instead of picking new work. Worktree-native: the halt row records the kept worktree path (N4.3), so N2.5 re-enters via `EnterWorktree path=…` rather than minting a new one. Multiple → EXIT for a human to pick.

## 3. Hard rules (binding)

**Unconditional hard stops (EXIT, never bypass):** cloud migration push (`{{profile.hard_stops.cloud_migration_*}}`), `npm install/uninstall`, edits to `~/.claude/settings.json`/`~/.claude.json`/any secrets file, `DROP`/`TRUNCATE`/unbounded `DELETE` in a migration, force-push/rebase on a shared branch, `--no-verify`.

**Dependencies are human-only by construction.** No seat inside a run can add a package: agents are banned from `npm install/uninstall` (the `subagent-templates.md` common preamble), the orchestrator is banned from main-seat `package.json`/`package-lock.json` writes (the product-write hook — see below), and `npm install/uninstall` is an unconditional hard stop (above). This is deliberate — an added dependency is unreviewed code entering the tree, so a human adds it in the open, never an autonomous run. **Consequence:** a ticket needing a package the target does not already have CANNOT be completed autonomously and MUST NOT dead-end into `blocked.md` mid-implementation. Instead the run catches the missing package at **E1** and HALTs EARLY with the exact package name + reason. **Recovery (human-preinstall-then-rerun):** a human runs `npm install <pkg>` (or `-D <pkg>`) in `{{profile.target.repo_path}}`, commits `package.json` + the lockfile on `{{profile.target.default_git_branch}}`, then re-launches `/run-build`; the dep is now present, E1's preflight passes, and the run proceeds normally. Encoded at E1 (preflight) + its same-rule E2/E3 backstop.

**Token / push rules:** no per-PR token cap (coarse per-iteration accounting → the night tally → a 10M/night backstop, §2 + N4); `git push` to a feature branch is allowed; push-to-default + `gh pr merge` are HELD for human (`[LAYER: auto-merge]`); concurrency cap 3; every spawn pins `model:` (hook-enforced); NO AI attribution trailer on commits (hook + setting).

**Orchestrator does NOT hand-write product code — NOR its own CI/gates** during a run (the product-write hook BLOCKS any main-seat Write/Edit to `src/`/`e2e/`/`supabase/migrations/`/`.github/**`/`.gates.conf`/`package.json`/`package-lock.json` while the sentinel is armed). The orchestrator writes only **state/plans/docs** — NOT product code and NOT CI config. *(Why `.github/**`+`.gates.conf` are banned: a main-seat run could otherwise keep a required check's NAME while gutting its BODY (`run: exit 0`) → a green check that asserts nothing but satisfies branch protection AND the pre-PR re-derive. Same class for `package.json`/lockfile = unreviewed supply-chain. Forcing these through a spawned agent lands them in the diff + AI review, and E1 auto-flags them HIGH.)* **Corollary:** gate *commands* that mutate product files — notably `types_regen` (`… > src/types/database.ts`) and its convenience-block re-append, authoring `e2e/**` tests, and any CI-workflow / `.gates.conf` / dependency edit — are done by a SPAWNED agent, NEVER by the orchestrator. The orchestrator re-verifies with READ-ONLY checks (exit codes, `git diff` inspection). Do NOT run `types_regen` (or a workflow/`.gates.conf` rewrite) as a main-seat Bash redirect either — it is a hook bypass and violates this rule even though the hook can't see it; delegate it, and the end-of-run isolation sweep is the named backstop for Bash-mediated writes.

**§VERIFICATION PROTOCOL (non-delegable):** the orchestrator re-runs every gate itself; a sub-agent's "green" is a hypothesis until the orchestrator's own tool call confirms it. Quantified divergence detection (agent claims 14/14, orchestrator sees 13/14 → integrity strike); 3 strikes → HALT; PNG-truth (never trust a screenshot's existence as proof of content). Re-verification uses read-only commands per the corollary above.

## 6. NIGHT DRIVER (N0–N4)

### N0 — Preflight: profile, mutex, night, git, tracker (order matters)
1. **Kill switch:** `ls ~/autopilot/.overnight-stop` present → EXIT clean.
2. **Load profile FIRST:** read `~/autopilot/profiles/<name>.yml` in full (this is also the start of the E0 cache anchor — E0 does NOT re-read it, it re-anchors). Absent → EXIT `[run-build] unknown profile` (`[LAYER: derive-profile]`). Any `TODO:` field blocks the phase that needs it.
3. **Mutex via the sentinel PID:** `<gates-dir>/autopilot-run-active.sh check` — if it returns 0, a live sibling session already holds the run (its PID is alive) → EXIT `[run-build] another iteration in flight`. (A stale sentinel from a dead prior session returns non-zero via the PID guard — proceed.)
4. **Night adopt-or-mint:** read `~/autopilot/state/current-night.json`. If present AND `now - night_start < 8h` → adopt its `run_id`. **Zero-work re-mint (human state-surgery must never be the mechanism):** on adopt, first check whether this night has done ANY real work — is `state/<run-id>/*/progress.yml` non-empty (any unit reached the engine), OR `state/<run-id>/held-pr.json` present (E6 writes it night-level, not per-unit), OR `loop-state.json.tokens_used_total > 0`? If ALL are empty/zero (every prior iteration HALT-EXITed at preflight before a unit was picked — e.g. tracker-dark or a capability-probe failure), the recorded `night_start` is dead clock that would eat the 8h window → **re-mint the clock in place**: set `night_start = now` in BOTH `current-night.json` and `loop-state.json` (KEEP the same `run_id` so night-scoped state stays continuous) and append `night-log.md` `[N0] zero-work night — re-minted night_start (preflight-only prior iterations; no unit had started)`. A night that has done ANY unit work is NEVER re-minted (its clock is real). Else mint `RUN_ID="{{profile.profile_name}}-$(date -u +%Y%m%dT%H%MZ)"`, write `current-night.json` `{run_id:RUN_ID, night_start:now}`, and init `state/<RUN_ID>/loop-state.json` = `{night_start:now, tokens_used_total:0, no_progress_count:0, night_backstop:10000000}`. `mkdir -p state/<RUN_ID>`.
5. **Arm sentinel:** `<gates-dir>/autopilot-run-active.sh start "$RUN_ID"` (arms the run-scoped gates). **Do NOT tear this down at EXIT** — the Stop hook runs the end-of-run audit while it is still armed, and the PID+TTL guard clears it at session death. Explicit teardown before the session stops disarms the audit. **Immediately VERIFY the arm took (sentinel-arm canary):** run `<gates-dir>/autopilot-run-active.sh check-owner` right after `start` — assert **`check-owner`, NOT plain `check`**: the product-write ban AND the Stop-hook end-of-run audit both gate on `check-owner` (process-ownership), so a passing plain `check` does NOT prove either load-bearing gate will fire — a green plain-`check` canary is itself a silent fail-open. `check-owner` is strictly stronger (subsumes `check`) and tests the exact predicate the gates use; it MUST return 0. Non-zero → the sentinel did NOT arm AS OWNED (the gates would silently never fire): its recorded owner-pid already looks dead or is not an ancestor of this process, which happens when the run is launched via a transient/daemon wrapper whose parent-pid is not the durable session. → **HALT-EXIT** `[sentinel-arm] arm canary failed — the launch broke the parent-pid invariant; relaunch with the durable session as the Bash parent, or run the end-of-run audit manually`. Do NOT proceed to N1. *(A topology-independent fix — sentinel liveness via a per-phase heartbeat file + a session/transcript-id ownership key instead of the parent-pid — is a tracked design task; a naive pid change regresses either the cross-session owner-guard or the product-write guard against later human sessions.)*
6. **Git preflight (worktree-native):** `git -C {{profile.target.repo_path}} fetch origin {{profile.target.default_git_branch}}` (read-only; ensures the fresh worktree at N2.5 branches off today's default). The main checkout's branch/dirty state **no longer blocks the run** — the unit runs in a fresh worktree off `origin/<default>`, so a human's WIP can sit in the main checkout untouched. Emit ONE `WARN` to `night-log.md` if the main checkout is not on `{{profile.target.default_git_branch}}` (informational only). The "don't branch off iteration N-1's feature branch" invariant is enforced structurally by N2.5 (fresh worktree off origin) + its `HEAD == origin/<default>` assert.
7. **Stop checks against the NIGHT tally** (read `state/<run-id>/loop-state.json`): night wall-clock `now - night_start ≥ 8h` → EXIT clean; `tokens_used_total ≥ 10,000,000` → EXIT clean (runaway backstop; a healthy night never nears it); `no_progress_count ≥ 2` → EXIT clean "no progress".
8. **TRACKER PROBE (mechanical, every run):** assert WRITE-CAPABLE Linear tools exist in THIS session's surface (not merely `authenticate`/`complete_authentication`). Resolve + log in `night-log.md`:
   - **LINEAR-LIVE** — write tools present AND a cheap read (`list_issues` on `{{profile.linear.project}}`) succeeds → track to Linear.
   - **SURFACE-STALE** — only auth tools exposed BUT a prior successful Linear write is on record (the global `tracker-debt.yml` has any `reconciled_at` set, or night history shows one) → fallback + debt, flag `SURFACE-STALE (refresh likely clears)`.
   - **LINEAR-DARK** — auth-only tools AND no prior write on record, OR the read returns an auth error → fallback + debt, flag `LINEAR-DARK (re-auth needed)`.
   - **Transient:** write tools present but the read fails with a NON-auth error (network) → retry once → on second failure treat as `LINEAR-DARK-with-reason:<err>`.
   Never assert the account is broken. Do NOT refresh the MCP surface while agents run — refresh between iterations, at N0.
   **HALT floor:** Linear not-live AND GitHub unreachable → EXIT (a run that can't record what it did does not run).
9. **Deep-reviewer routability probe (once per night):** the run env can silently deliver a `model:<deep-review-model>` spawn as a fallback model, which would invalidate the §E5.6 gate. If `state/<run-id>/loop-state.json` already carries a `reviewer_routable` field (a prior iteration of THIS night already probed) → adopt it, do NOT re-spawn. Else spawn ONE tiny throwaway agent (`model:` the deep-review model, label `reviewer-probe`, trivial prompt `reply OK`) and inspect its transcript model field: the intended model → write `reviewer_routable: true`; anything else (silently a fallback) → write `reviewer_routable: false` AND emit a `WARN` to `night-log.md` (`[N0] deep-review model NOT routable this session — §E5.6 will flagged-HOLD HIGH/Medium units, gate un-run`). Cheap; runs once before any E0–E5 spend; §E5.6 step 5 reads this.
10. **Writability:** the background-isolation guard is *resolved* by entering a per-unit worktree at **§N2.5** — post-`EnterWorktree`, the guard permits (indeed redirects) writes under the worktree, so background and foreground behave identically. The write-probe runs at **N2.5 step 4** (post-worktree), where a BLOCK is an unexpected platform condition → HALT.

### N2 — Pick work (ticket-mode)
**Operator selection wins over the rubric (`--unit`):** if the `/run-build` launch args (`$ARGUMENTS`) carry an explicit selection, honor it instead of the rubric — `--unit {{profile.linear.ticket_id_prefix}}-<id1>+{{profile.linear.ticket_id_prefix}}-<id2>[+…]` → build exactly that themed bundle (the operator-directed path of the bundle rules below); `--ticket {{profile.linear.ticket_id_prefix}}-<id>` → build exactly that one ticket (the `/run-build` command surfaces these args). No explicit selection → pick by the rubric below. **Operator-selection edge cases:** a single-id `--unit <id>` (no `+`) = a SOLO unit (not a bundle); `--ticket` AND `--unit` both present → **HALT** `[N2] ambiguous selection — both --ticket and --unit given`; an operator-named member NOT in Todo/Backlog (already Done/In-Progress/closed) → **refuse** `[N2] --unit member <id> is <status>, not buildable` (operator override does NOT bypass the buildable-status check — the N2.5 stale-branch probe only guards members whose old branch still exists).
Priority rubric: next `{{profile.linear.project}}` item that is Todo/Backlog, no unmet `Blocked by`. **Tracker-authoritative scope:** fetch the full ticket body (`get_issue`) before scoping — never extrapolate from a one-liner.
- **Linear dark → work source:** use the profile's `{{profile.artifacts.linear_mapping_doc}}` "Active workstream" table (Backlog rows, no unmet Blocked-by), applying the SAME no-extrapolation caveat (the mapping row is a pointer, not the scope — if the full ticket body is unreachable, do NOT invent scope; report and EXIT). If neither Linear nor the mapping doc yields an unblocked unit → EXIT clean `[run-build] queue empty`.
- `[LAYER: theme-mode]` would read an approved `slices.yml` here instead.

**Themed bundle — a unit MAY be 1..N tickets (§N3), bound ONLY on a machine-checkable signal (never semantic-similarity judgement):**
- **operator-directed:** a launch arg `--unit {{profile.linear.ticket_id_prefix}}-<id1>+{{profile.linear.ticket_id_prefix}}-<id2>[+…]` (or a queue row) names the members explicitly; OR
- **unattended:** members share the SAME Linear parent, OR the same declared theme label, OR form a declared `Blocked-by` chain — bundling is the ONLY way to ship dependent tickets in one night (N2.5 always branches off `origin/{{profile.target.default_git_branch}}`, so a second ticket cannot otherwise stack on an unmerged first). No signal → the unit stays solo.

Fetch EVERY member's full `get_issue` body (same no-extrapolation rule, per member). Compute `unit_id = {{profile.linear.ticket_id_prefix}}-<id1>-<id2>[-…]` (members in ticket order) and write `state/<run-id>/<unit-id>/unit.yml` `{members, primary:<lead ticket>, rationale, directed_by:<operator|signal>, bundle_signal}`. **Bundle-size cap — refuse over-cap bundles and fall back to one-ticket-per-unit picking:** projected diff >~15 files, OR >2 architectural areas, OR a migration+engine+UI mix → refuse HERE (log `[N2] bundle refused — over size cap; picking members solo`) and pick the members as solo units by the normal rubric. (A `Blocked-by`-chain bundle that is refused can only ship its HEAD this night — the blocked members wait for the next night after it merges, since N2.5 branches off `origin/<default>`.)

### N2.5 — Enter the unit worktree (worktree-native; supersedes E2 branch creation)
The whole unit runs in a per-unit git worktree, so **background and foreground runs are identical** and the **main checkout is never mutated**. Night state lives under `~/autopilot/state/` (outside the launch root — guard-exempt), so entering here (after the unit is picked) is safe and queue-empty/EXIT-early iterations never create a worktree.

1. **Held-PR / stale-branch check FIRST, from the main checkout (read-only) — LOOPED over EVERY member id:** a bundle's branch is named for the whole unit (`{{profile.linear.ticket_id_prefix}}-<id1>-<id2>-…`), so a single-id `git ls-remote --heads origin "…-<id>-*"` MISSES it — a later run that picks a bundled id alone would not see the existing compound branch (a real duplicate-work hazard under Linear-dark). So for EACH member id in `unit.yml` (just the one, for a solo unit), test every `git ls-remote --heads origin` branch with a PRECISE rule (avoids both the substring trap and a numeric-token collision in an unrelated ticket's kebab): the branch matches member id `<m>` iff (a) it begins with `{{profile.linear.ticket_id_prefix}}-`, AND (b) `<m>` (the bare numeric id) appears as a WHOLE `-`-delimited segment within the **leading id-run** of the branch name — the maximal run of all-numeric segments immediately after the prefix, before the first non-numeric (kebab) segment. So `{{profile.linear.ticket_id_prefix}}-62-51-fuel-audit` → id-run {62,51} matches member 62 or 51; `{{profile.linear.ticket_id_prefix}}-702-upgrade-to-nextjs-16` → id-run {702} does NOT match member 16 (the `16` is a kebab word, not in the id-run); and member `5` never matches `51` (whole-segment). ANY matched remote branch with an OPEN held PR → **EXIT clean** `[run-build] unit (or member <id>) already has a held PR #<N>`; a matched branch with NO PR → **HALT** for human cleanup (prior failed run).
2. **Enter / create the worktree:**
   - If `.claude/worktrees/<branch>` exists AND the N0 resume-scan matched this unit → `EnterWorktree path=.claude/worktrees/<branch>` (re-enter to resume).
   - Else if it exists but was NOT resume-matched → **WARN + skip the unit (move on) — do NOT auto-remove; cleanup stays human-only**: append a `WARN` line `[run-build] stale worktree for <branch> — human cleanup needed; skipping unit` to `night-log.md`, leave the worktree on disk untouched for morning cleanup, and skip this unit per §9 ("never burn the night on one unit") so the night moves on to other units — do NOT HALT and do NOT `worktree remove` it. (The system can strand a worktree it created itself — self-inflicted debris must WARN-and-skip, never dead-end the night.)
   - Else → `EnterWorktree name=<branch>` where `<branch>` = `{{profile.linear.ticket_id_prefix}}-<id>-<short-kebab>` for a solo unit, or **`{{profile.linear.ticket_id_prefix}}-<id1>-<id2>[-…]-<short-kebab>` for a bundle** (the same compound id-run the step-1 probe matches on, then one shared kebab) — creates worktree `.claude/worktrees/<branch>` based off `origin/{{profile.target.default_git_branch}}` (`worktree.baseRef` default `fresh`). **Keep the whole branch name within the 64-char/charset limit — for a many-member bundle, shorten the shared `<short-kebab>` (the id-run is what the probe and PR need, not a long slug).**
   - **Rename the branch (MANDATORY):** `EnterWorktree name=X` puts the worktree on branch **`worktree-X`** (the harness prepends `worktree-`; name ≠ branch). So immediately, INSIDE the worktree: `git checkout -b <branch>` (the `<prefix>-<id>-<short>` name the convention + PR require) then `git branch -D worktree-<branch>` (now unreferenced). Skip this on the resume path (`EnterWorktree path=…` re-enters the already-renamed branch).
3. **Verify, don't assume (3 read-only checks; any mismatch → HALT):** cwd is now the worktree; `git rev-parse HEAD` == `git rev-parse origin/{{profile.target.default_git_branch}}` (catches a `worktree.baseRef=head` override or a stale fetch, without ever reading a settings file); and AFTER the rename, `git rev-parse --abbrev-ref HEAD` == `<branch>` (the convention name, not `worktree-…`).
4. **Writability probe:** spawn ONE throwaway agent (`model: haiku`, label `write-probe`) that uses the **Write tool** (NOT Bash — Bash bypasses the platform guard) to create `<worktree>/artifacts/.write-probe-<run-id>`; read the tool result. **PASS** (file on disk, no `is_error`) → `rm` it, proceed (the expected outcome post-worktree). **BLOCK** → **HALT** with the raw error (unexpected platform condition: the worktree should always be writable).
5. **Write the isolation manifest** `~/autopilot/state/run-worktrees.json` (the file the end-of-run audit prefers):
   `[{"path":"<worktree-abs>","expected_globs":["src/**","e2e/**","supabase/migrations/**","artifacts/**"]},{"path":"<main-checkout-abs>","expected_globs":[]}]`
6. **Set `{{active_root}}` = `<worktree-abs>`** for the rest of the iteration: every gate command, every sub-agent brief path, and every repo-relative reference resolves against it (subagent-templates.md path roots are `{{active_root}}`-slotted).
7. **ECHO worktree mode to the operator (the standing "am I in a worktree?" signal):** emit ONE visible line to the session output AND append the same to `night-log.md`: `[worktree] <ticket-id> → <worktree-abs> on branch <branch> (main checkout untouched)`. Attended runs rely on this to confirm the unit is worktree-isolated. If any E-phase runs without this line having been emitted first, N2.5 was skipped — that is a bug → HALT.

### N3 — Execute ONE unit via ENGINE (E0–E7)
**One unit = one branch = one PR = one state dir = one ledger — where a unit is 1..N tickets (§N2 bundle).** The invariant is unchanged (a unit still maps to exactly ONE PR); only the definition of *unit* widens: a solo unit is its single ticket; a bundle is its `unit.yml` members under one `unit_id`. A mixed bundle's **risk tier = `max()` of its members' tiers** (E1), and the **trivial-scope skip is NEVER available to a bundle** (>1 member ⇒ never trivial). Blast radius, named honestly: one PR = one revert unit — so bundle only genuinely-coupled tickets, give each member its own commit series, and carry a commits→tickets map in the PR body.

### N4 — Close
1. **Token tally (named source):** sum `<usage>total_tokens: N` from every sub-agent reply THIS iteration; add to `state/<run-id>/loop-state.json` `tokens_used_total` (one summed write at close — no per-reply jq).
2. **No-progress:** if this iteration reached PR-ready (a held PR opened) → set `no_progress_count = 0`; else `+= 1`.
3. **Exit the worktree (worktree-native):** PR-ready → **do NOT use `ExitWorktree action:remove`**: after N2.5's branch rename (`worktree-<name>` → `<branch>`, old name deleted) the tool still tracks the deleted `worktree-<name>` and false-flags the commit as *permanent-loss* on removal — observed leaving stray worktrees on disk. Instead: **(a)** confirm the commit is safe on origin (`git -C {{profile.target.repo_path}} ls-remote --heads origin <branch>` shows the pushed branch); **(b)** `ExitWorktree action:keep` to return to the main checkout WITHOUT tripping the permanent-loss flag; **(c)** `git -C {{profile.target.repo_path}} worktree remove --force .claude/worktrees/<branch>` (git-native cleanup — the pushed feature branch + held PR persist on origin). HALT mid-unit → `ExitWorktree action:keep` AND record the worktree path in the unit's `progress.yml` halt row so the resume-scan can re-enter it by path (§N2.5 step 2). The **main checkout is NEVER checked out, pulled, or otherwise mutated by the run** — the next iteration's N2.5 branches a fresh worktree off `origin/<default>` regardless. List any kept worktrees in the `night-log.md` append (morning cleanup).
4. Append the `night-log.md` line (run-id · unit · tracker state · debt count · PR/HOLD status).
5. **EXIT clean WITHOUT tearing down the sentinel** — the Stop hook runs the end-of-run audit (isolation sweep + model watchdog) while the sentinel is still armed, surfaces any finding once, and the PID+TTL guard clears the sentinel at session death.
6. **Regenerate, never edit, a durable/verified audit copy:** if a copy of the audit report is preserved (e.g. under `state/<run-id>/` to survive a shared-path clobber), that copy is produced ONLY by re-running the end-of-run audit and overwriting the file wholesale — NEVER by `sed`/manual patch/partial Edit on an existing report. A report titled "verified"/"durable" that has been hand-patched contradicts its own content even when the patched claim is factually true; if a value in it is wrong or stale, re-run the audit, don't patch the number.

## 7. ENGINE (E0–E7, per unit)

**Bracketing rule (binding, no exceptions):** every phase the engine enters — every N-phase (N0–N4) and every E-phase (E0–E7) alike — is bracketed by `progress.yml` appends: a `start` row FIRST, a `complete|halt|exit` row LAST. This binds ADOPTED phases too (work reused from a prior/aborted run, e.g. E0–E2 context/spec adopted by path instead of re-run): still write a `start` row (annotated `adopted_from: <run-id>`) THEN a `complete` row — **never complete-only**. A phase recorded complete-only (missing its `start`) is the SAME violation as one missing its close: `progress.yml` must NEVER be complete-only, because the resume/crash-recovery scan (§2) reads the bracket to tell a genuinely-fresh phase from a crashed one. A phase ending without both rows = corrupted run → HALT-EXIT.

### E0 — Bootstrap (cache anchor — deliberately small)
Single parallel batch, exact order (the prompt-cache anchor — do NOT re-read later): `~/CLAUDE.md`, `{{profile.target.repo_path}}/CLAUDE.md`; `~/autopilot/subagent-templates.md`; `~/autopilot/state/<run-id>/theme-ledger.md` (if present); `{{profile.memory.target_memory_index}}` (index only; lazy-load gotchas); `tail -50 ~/autopilot/state/<run-id>/night-log.md`. The profile was read at N0.2 — re-anchor from context, do NOT re-read. Ticket body + spec are read in E1/E2, not here.

### E1 — Scope + trivial-scope skip (inlined)
`get_issue` full body (tracker-authoritative — for a bundle, all members' bodies were already fetched at N2; scope the combined change). Classify risk tier from `{{profile.sensitive_surfaces}}` + generic globs (`auth*`/`rls*`/`payment*`/`migrations/**`) → HIGH; **any diff touching `.github/workflows/**` / `.gates.conf` / CI config → automatically HIGH (a run editing the gate it is judged by is the highest-trust-impact change there is)**; else define the tier MECHANICALLY: **LOW** iff it satisfies the trivial-scope-skip shape below (≤2 files, ≤~20 net lines, no new exported symbol, none of the forbidden categories); **everything else that is not HIGH → MEDIUM**. **Bundle unit: scope the COMBINED change over all `unit.yml` members and take tier = `max()` over members — any member HIGH ⇒ the unit is HIGH; the trivial-scope skip is unavailable whenever `unit.yml` lists >1 member (a bundle is never trivial).** This tier is estimated pre-code from the ticket scope and DECIDES whether §E5.6's deep-review gate runs, so §E5.6 step 1 RECOMPUTES it from the actual diff. Log the tier decision as one auditable line. **Trivial-scope skip (INLINED):** skip spec ceremony only if ALL hold — (a) ≤2 files, (b) ≤~20 net lines, (c) no new exported symbol, (d) NONE of the forbidden categories: migrations, auth/RLS, money/pricing, or interactive dialogs/downloads. Any forbidden category → full spec, never trivial. Log the skip decision as one auditable line.

**Dependency preflight — HALT EARLY with the exact package, don't dead-end mid-implementation (deps are human-only, §3):** as part of scoping, determine which packages this ticket requires and check each, read-only, against `dependencies`+`devDependencies` in `{{profile.target.repo_path}}/package.json` (the worktree bases off the same default branch, so its `package.json` is identical). **If ANY required package is absent** → do NOT proceed to E2/E3: no seat inside a run can `npm install` it (§3), so implementation could only dead-end mid-wave into `blocked.md`. Instead HALT the unit HERE and surface a precise, actionable message — also written to `state/<run-id>/<ticket-id>/blocked.md` (§8 rule 8) and as an E1 `halt` row in `progress.yml` (§7 bracketing rule):
  `[run-build] HALT E1 — ticket <id> needs dependency <pkg> (<why the ticket needs it>), absent from package.json. Dependencies are human-only (supply-chain safety, §3). RECOVERY: a human runs "npm install <pkg>" (or "-D <pkg>") in {{profile.target.repo_path}}, commits package.json + the lockfile on {{profile.target.default_git_branch}}, then re-launches /run-build — on the re-run <pkg> is present, this preflight passes, and the run proceeds.`
  Name EVERY missing package (one line each), not just the first. **Same rule at any later phase:** if a required package first surfaces at E2 (the spec enumerates a new import) or E3 (an implementer reports a missing module), apply the IDENTICAL early-HALT-with-package-name-and-recovery — human-preinstall-then-rerun is the disposition for a missing dependency at every phase; E1 is simply where it SHOULD be caught first, before any implementation work is spent.

### E2 — Spec (branch + worktree now created at §N2.5)
The held-PR / stale-branch `git ls-remote` check AND branch creation happen at **§N2.5** (worktree-native) — E2 no longer creates the branch. Running INSIDE the worktree (`{{active_root}}`): spawn Template A explorer (**haiku**) → context inventory; Template B architect (**opus**) → binding spec at `state/<run-id>/<unit-id>/spec.md` — **for a bundle, ONE spec covering the combined change with a per-member acceptance section (each `unit.yml` member its own criteria).** If the spec plans a deliberate rewrite of an existing route/component, set a `planned_rewrite: [paths]` flag in the spec (the diff-triggered inventory itself runs at E4).

### E3 — Implementation waves (inlined structure)
Waves, each a pinned **sonnet** implementer (Template C/D), orchestrator re-verifies after EACH wave with READ-ONLY checks:
- **Wave A — DB (only if migration needed):** the IMPLEMENTER runs the migration + `{{profile.gates.lint_migrations.cmd}}` + `{{profile.gates.migration_up_local.cmd}}` + **`types_regen` AND the convenience-block re-append** (both mutate `src/` → they are the agent's job, NEVER the orchestrator's). Orchestrator re-verifies read-only: `{{profile.gates.tsc.cmd}}` exit 0, and `git diff --stat src/types/database.ts` shows the regen + a `grep` confirms the convenience block is present. Live DB gates are serialized (shared-stack rule).
- **Wave B — backend / Wave C — frontend:** pinned sonnet implementers; orchestrator re-verifies `tsc`/`eslint` exit codes itself. Per-gate fix cap 3, global 5. The orchestrator never touches `src/` (hook-enforced).

### E4 — Review + arbitration ledger + rewrite-inventory
1. **NO CodeRabbit at E4.** CodeRabbit is the one *premium* reviewer, so it gets exactly ONE CLI pass per run, positioned on the FINAL, verified, committed diff where it catches what the standard reviewers miss — never duplicating them, never reviewing un-verified code. That single pass is the **pre-PR-gate CR call at E6 step 2** (`--type committed`, after E5 verification; it is the *only* `coderabbit review` the run runs — the pre-PR evidence hook re-derives only the `rederive:true` check, not CR, so it fires once). Do **not** run `coderabbit review` here at E4. CI-arriving CR comments are still harvested as data, never a new loop entry point.
2. Spawn code-reviewer + design-reviewer (grumpy-designer), pinned per §5. **Rewrite feature-inventory (diff-triggered):** compute `git diff origin/{{profile.target.default_git_branch}}...HEAD --stat`; for any DELETED / renamed / >50%-rewritten file, enumerate the OLD shape's features (`git show <base>:<path>`) and add to the reviewer brief a line-item requiring each to be confirmed preserved or intentionally dropped-in-spec. (This is the silent-regression control — a rewrite can drop a feature another PR added days earlier; it can only fire here, where the diff exists.)
3. **Arbitration ledger `arbitration-<ticket-id>.yml`:** every finding gets ONE disposition `fix | reject | defer` + decider + one-line rationale. C/H correctness/security → `fix`; Medium → `fix` if ≤10-line diff else `defer`; Low/style/nitpick → `defer`. A `reject` needs evidence (test run / file:line disproof), never "we disagree stylistically". The E4 (standard-reviewer) findings are all dispositioned here; the single CodeRabbit CLI pass runs later, at E6 step 2, and **its** findings are dispositioned into this same ledger before `gh pr create` (E6 step 2a). No PR is CREATED with an undispositioned finding.
4. **Contested C/H** → record it in the ledger and **HALT for human** with the disagreement stated.
5. **Max 2 fix rounds.** Round 1 fixes the `fix` set; round 2 handles only NEW C/H from round 1; remaining new C/H after round 2 → HALT. Zero-findings is explicitly NOT a merge criterion. Fix agents are FRESH pinned spawns each round — never SendMessage-resumed (resume-leak; a hook blocks it under an expensive driver anyway).

### E5 — Test authoring + verification + slop gate
- **E5.0 — Spawn ONE test-engineer (Template E, sonnet)** to author `e2e/pr<N>-verify*.spec.ts` (the orchestrator cannot write `e2e/**` itself — hook). Required BEFORE the playwright gate runs.
- **§VERIFICATION PROTOCOL re-runs (non-delegable):** orchestrator runs the full gate table itself; quantified-divergence checked; `{{profile.gates.playwright.cmd}}` run by the orchestrator.
- **Seam tests per interactive class** (Template E 2.5, in the E0-read templates): downloads assert magic bytes; form submits assert DB state via SELECT; access-control asserts both branches; navigation asserts destination render. Presence-only assertions insufficient.
- `[LAYER: full-slop-gate]`: tests-fail-on-main, criterion→evidence table, wiring check, placeholder scan — NOT built. LOG that these are deferred (honest, not a claimed pass).

### E5.6 — Deep pre-PR review (the muscle before the net)
**Applies to HIGH + Medium risk-tier units only — by the tier RECOMPUTED from the actual diff at step 1, not E1's pre-code guess.** LOW/trivial → SKIP (log `[E5.6] deep-review gate skipped — LOW tier` to `theme-ledger.md`) and go straight to E6. The reserved security/quality pass runs HERE, before the PR exists, so a Critical/High/Medium issue is caught by the **muscle**, not left for CodeRabbit (the **net** at E6.5) or the human.

1. **Recompute the risk tier from the ACTUAL diff, THEN gate.** Commit first so the tree is clean (the same commit E6 step 1 would make). `git diff --name-only origin/{{profile.target.default_git_branch}}...HEAD` matched against E1's sensitive/HIGH path patterns (`{{profile.sensitive_surfaces}}` + `auth*`/`rls*`/`payment*`/`migrations/**` + `.github/workflows/**`/`.gates.conf`). E1's tier was a pre-code guess — this recomputed tier is authoritative: if the diff touches a sensitive path E1 didn't anticipate → **upgrade to HIGH and run the gate even if E1 said LOW**; if the recomputed tier is LOW/trivial → SKIP (log as above) and go to E6. **Target** for the review = the E5-verified, green diff `git diff origin/{{profile.target.default_git_branch}}...HEAD`.
2. **Spawn ONE deep-review agent** (`model:` the deep-review model, ultrathink, label `deep-prepr`). Keep the prompt **PLAIN** — send the diff and ask for a findings report; do NOT wrap it in adversarial "find the exploit / attack it" framing (it reviews *worse* that way AND can trip a sensitive-content classifier → the review gets degraded/blocked). The brief is literally: *"ultrathink. Review this diff and report findings — correctness, security/authorization, data exposure, tests, edge cases, AND whether the code is clean, efficient, and not over-engineered (flag unnecessary complexity, dead abstraction, bloat). Severity + file:line + the concrete scenario per finding. Do not write code. End with a one-line verdict."* Pass the diff **plus the unit's `state/<run-id>/<unit-id>/spec.md` and the ticket body(ies) — for a bundle, EVERY `unit.yml` member's ticket body, not just the primary's (so a member's spec-intended behaviour isn't flagged)** by file path (never pasted inline) — the spec + ticket give the reviewer the scope context so it does NOT flag spec-intended or spec-deferred behaviour as a finding; keep the prompt PLAIN, just also hand it the spec + ticket, add no adversarial framing. **If N0 recorded `reviewer_routable: false`, do NOT spawn — go straight to the step 5 degrade (flagged-HOLD).**
3. **Disposition into `arbitration-<ticket-id>.yml`:** Critical / High → `fix` (strict). **Medium → `fix`, OR `defer-with-evidence` when the spec/ticket EXPLICITLY defers the item** (a documented scope-boundary Medium may be deferred-with-evidence, not forced to fix-or-HALT). Low / Info → `defer-with-reason` (recorded in the ledger + carried in the E6 PR body). "Over-engineered / could be simpler" at Medium+ is a valid `fix` — **clean, efficient, non-over-engineered code is the bar, not merely correct code.**
4. **Fix loop (bounded, mirrors E4):** fresh pinned **sonnet** implementers fix the C/H/M set; orchestrator re-verifies with the E5 gate table (non-delegable); **max 2 rounds**; then re-run the deep-review agent ONCE on the fix as a confirming pass — the confirming pass surfaces/handles only **NEW** C/H/M introduced by the fixes (mirror E4's round-2 "only NEW C/H" rule), so a pre-existing Low it re-rates Medium does NOT thrash toward HALT. Remaining C/H/M after round 2, or a **contested** finding → **HALT for human** (`blocked.md` + night-log) — never open the PR past an unresolved C/H/M.
5. **Routability check + reviewed-sha record (cheap, non-fabrication).** The `deep-prepr` transcript's model field MUST be the intended deep-review model throughout — the run env can silently deliver the spawn as a fallback model, which invalidates the gate (N0 step 9's `reviewer_routable` probe is the early signal).
   - **On the intended model** (and N0 `reviewer_routable != false`): record the verdict + dispositions **AND the exact reviewed HEAD sha** (`git rev-parse HEAD`) for the E6 PR body; proceed to E6 only when no unresolved C/H/M remains.
   - **NOT on the intended model** (the spawn was delivered as a fallback, or N0 recorded `reviewer_routable: false`): **ONE retry** of the `deep-prepr` spawn; if it is STILL not the intended model → **DEGRADE**. Do **NOT** HALT the night (§9 — never burn the night on one unit) and do **NOT** count the fallback-delivered review as the deep-review gate: **SKIP** the gate for this unit and **HOLD** the PR carrying the loud flag `[E5.6] deep-review gate NOT RUN — reviewer model unroutable this session; HIGH/Medium unit unreviewed` in the PR body + `night-log.md` + the morning report. Never silently substitute a fallback model for the gate.

### E6 — PR → mark-ready → FULL CI → HOLD
All commands run INSIDE the worktree (`{{active_root}}`): the pre-PR gate resolves its root via `git rev-parse --show-toplevel` and reads the tracked `.gates.conf` from the worktree; the pre-PR evidence hook resolves the repo from the hook payload cwd, so evidence lands and is checked at the worktree root; `git push`/`gh pr create` are worktree-normal (the branch already exists on origin's namespace via the pushed feature branch).
1. `git add <explicit-paths>` (never `git add .`); verify no secrets/oversized binaries; commit (NO attribution trailer — hook-enforced); `git push -u origin <branch>`.
2. Pre-PR gate suite `{{profile.gates.pre_pr_gate.cmd}}` → exit 0 required (independently re-derived by the pre-PR evidence hook on `gh pr create`).
2a. **Disposition the pre-PR gate's CodeRabbit findings (the run's SINGLE CR CLI pass).** The pre-PR gate ran `coderabbit review --agent --type committed --base main` on the verified committed diff (`.gates.conf`; the *only* `coderabbit review` the run runs). **CR-skip case:** if `.gate-evidence.json` shows the `coderabbit` check as `skipped:true` (the gate SKIPs — not fails — CR when `coderabbit doctor` is not "Signed in"), CR did NOT run at the CLI and there are NO CLI findings to triage — do NOT treat the skip as a clean pass: loudly log `[cr-cli] SKIPPED (coderabbit unauthed)` in the PR body + night-log, and lean on the E6.5 GitHub App review (a separate surface) as the CR gate. Otherwise (CR ran), triage every finding into `arbitration-<ticket-id>.yml`: C/H → fix (loop the implementer) + re-verify, Medium → fix-if-≤10-line else defer-with-evidence, Low/style → defer. A C/H fix re-commits → the pre-PR gate is re-run to regenerate evidence for the new HEAD (which re-reviews the fix — a legitimate confirming pass). Happy path (no C/H fix) = **exactly one** CR CLI pass. The E6.5 GitHub review is a DIFFERENT surface (the CodeRabbit App on the ready-flip), not a CLI call.
2b. **Minimum-review-floor for HIGH units (review layers can degrade independently and politely, and nothing notices when they ALL degrade at once).** Before `gh pr create`, for a unit whose §E5.6-recomputed tier is **HIGH**, COUNT the independent pre-PR review surfaces that ACTUALLY produced a review this run (not a skip/degrade): (1) the E4 code-reviewer, (2) the E4 grumpy design-reviewer (skips when the diff has no UI surface), (3) the §E5.6 deep-review gate (degrades when the reviewer model is unroutable), (4) the E6 step-2a CodeRabbit CLI pass (skips when coderabbit is unauthed). **≥2 ran → proceed to step 3 normally.** **Fewer than 2 ran → this HIGH change got only one real review and MUST NOT reach the morning as a normal PR.** This can only happen when the §E5.6 deep-review gate degraded (a routing HIGH unit always has ≥2: code-reviewer + deep-review), so it is the SAME terminal state as the §E5.6 degrade — a **HOLD-with-loud-flag, NOT a HALT**: still open the PR at step 3 (so the work is visible and the pushed branch is not stranded), but carry the loud flag `[E6] HIGH unit below minimum-review-floor — only <N>/4 review surfaces ran (<which>); REQUIRES human review, do NOT blind-merge` in the PR body + `night-log.md` + the morning report, and NEVER describe it as deep-review-cleared. (A HALT here would strand the just-pushed branch and let the resume-scan re-enter and re-HALT the unit — its thin-coverage condition is night-wide and will not change — pinning the night against §9 "never burn the night on one unit"; the HOLD leaves an open PR, so the resume-scan sees "branch + open PR" and correctly leaves it for the human.) (Medium/Low units: log the surface count as advisory, do NOT gate.)
3. `gh pr create --draft` with body per `{{profile.artifacts.pr_template}}` INCLUDING: the arbitration ledger, the Verification section with actual exit codes, **— for a BUNDLE, a per-member block (each `unit.yml` member: acceptance criteria → satisfying evidence) + a commits→tickets map (one PR = one revert unit) —** and — for **HIGH+Medium units** — the **deep pre-PR review verdict + dispositions from §E5.6**, worded as `Deep pre-PR review: CLEARED at <sha> (N findings dispositioned) — any later fix commits (CodeRabbit CLI / App reconcile) were reviewed by CodeRabbit only`, where `<sha>` is the reviewed HEAD sha recorded at §E5.6 step 5 (the claim is scoped to the reviewed sha, NOT the final merged sha). If a post-E5.6 fix commit (E6 step 2a CR CLI / E6.5 CR App reconcile) touches a sensitive glob → run ONE confirming deep-review re-pass on the new sha and update `<sha>`. If §E5.6 **degraded** (reviewer unroutable), carry its `[E5.6] deep-review gate NOT RUN …` flag here INSTEAD of a CLEARED line — and if step 2b's minimum-review-floor also failed (a HIGH unit with <2 surfaces run), carry its `[E6] …below minimum-review-floor…` flag alongside it (same held-PR-with-loud-flag terminal state, never a CLEARED line). For **LOW** units, `Deep pre-PR gate: N/A (LOW tier)`. The PR reaches the human already deep-reviewed (muscle), with CodeRabbit (net) to follow at ready-flip. Never present a HIGH-tier PR to the morning identically to a LOW one. **Draft-first ONLY so any draft-lane CI runs cheaply during the build — the PR is NEVER left draft (step 5).** **Then record the PR for the completion-gate: write `state/<run-id>/held-pr.json` = `{"number":<N>,"repo":"<owner/repo>"}` (state write, allowed). The completion-gate Stop hook reads this and BLOCKS run-close (+ writes a morning-visible night-log flag) unless the PR is one-click-mergeable — evaluated on the **LATEST run per required-check name** (so a superseded draft-lane `SKIPPED` does NOT false-trip it): it blocks while the PR is draft, while any required check's latest run is failed / `SKIPPED` / `NEUTRAL` / still-pending, while a required status-context (e.g. `CodeRabbit`) is non-success, or while CodeRabbit's latest *decision* is `CHANGES_REQUESTED`; it blocks once then night-log-flags. So the run can never silently close on a non-one-click-mergeable PR.**
4. Watch any draft-lane CI (`gh pr checks`) to green.
5. **MARK READY — `gh pr ready <N>` (NON-OPTIONAL, the linchpin).** A DRAFT PR is skipped by the security-spine CI **and never reviewed by CodeRabbit** — leaving it draft hands the human an ungated, unreviewed PR that they must un-draft themselves. Un-drafting FIRES the full gate. Configure your default branch's required checks + required PR review in your ruleset/branch protection (see `{{profile.ci.required_checks}}`); un-drafting fires the full required-check CI **and** any required review **and** any merge-block rules. Run `gh pr ready` from the **run session (box `gh` = PAT)** — **NEVER from inside GitHub Actions with `GITHUB_TOKEN`** (GitHub suppresses the triggered run → the required check stays `skipped` → the PR is mergeable UNGATED). Confirm `gh pr view <N> --json isDraft` == `false` before proceeding.
6. Watch the REAL, FULL CI on the now-ready PR: `gh pr checks` / `gh run list --commit <head-sha>` until complete (never local-green-only). **Evaluate the LATEST run per required-check name — a ready-flipped PR's raw `statusCheckRollup` still carries the superseded draft-lane `SKIPPED`/`CANCELLED` run right next to the ready-lane `SUCCESS`; that pair is EXPECTED, not the trap (`gh pr checks` already collapses to the effective per-name result — trust it). The LATEST run of every required check (`{{profile.ci.required_checks}}`) must be `SUCCESS`. Only if a required check's *latest* run is `SKIPPED`/`NEUTRAL` (the flip genuinely didn't fire it) → the ungated-merge trap → investigate, do NOT proceed.** → then **E6.5** (CodeRabbit reconcile — now that the PR is ready, CR will actually review).
7. **HOLD for human — at the RIGHT terminal state.** `[LAYER: auto-merge]`: do NOT `gh pr merge`, do NOT push to default. The held PR MUST be: **open + READY (not draft)** + **full CI green (no `SKIPPED` required check)** + **CodeRabbit reconciled (E6.5 — not `CHANGES_REQUESTED` with an unaddressed finding)** + ledger/risk in the body — so the human's ONLY action is *read the brief and click merge*. Migration-touching PRs are held regardless, always.

### E6.5 — Post-PR CodeRabbit reconciliation (check-green ≠ findings-addressed)
After the **E6 mark-ready flip** (step 5) + full-CI watch — CodeRabbit does **NOT** review drafts, so un-drafting is what triggers its review — CodeRabbit's **GitHub review** posts asynchronously — gate on the `coderabbitai` **review object's `.state` + its comments**, NEVER the check-rollup — and NOT the PR-level `reviewDecision`, which GitHub leaves EMPTY for *bot* reviews (an APPROVED CodeRabbit reads as blank there). (`CodeRabbit: pass` means it RAN, not that it APPROVED; the E6 step-2 pre-PR-gate `coderabbit review --type committed` CLI pass — the run's single CLI review — is a *different* surface from this GitHub App review). Poll `gh pr view <N> --json reviews` until a `coderabbitai` review lands after the head-sha push (poll ~5 min; if CR shows **activity** — a pending CodeRabbit status-context / partial inline comments — extend up to ~15 min before HOLD-PENDING, since a large diff / slow day can exceed 5 min on a PR that would've been clean by ~minute 7; the completion-gate Stop hook is the enforcement backstop either way); still pending at timeout → HOLD + loudly log `[cr-pr-review] PENDING — findings not dispositioned` in the PR body (never claim "CR clean"). Fetch every finding (`gh api …/pulls/<N>/comments` + review body) → disposition each into the arbitration ledger (`fix | defer-with-evidence | reject-with-evidence`) and update the PR body. **Severity gate:** Critical/High → fix (loop implementer, re-verify) or HALT-for-human; Medium → fix or defer-with-evidence; minor → fix-if-trivial else defer-with-reason. **A CR finding on the run's OWN change is not automatically valid** — if it flags a legitimate scope boundary (e.g. a behaviour change folded into a behaviour-preserving migration), the correct disposition may be to REVERT the change, not defer the finding. The run may NOT reach E7 while the PR is still a **DRAFT**, while any required check reads **`SKIPPED`** (E6 steps 5–6), while any CR PR-review comment is un-dispositioned, or while the latest review is `CHANGES_REQUESTED` with an unaddressed finding — i.e. the run may only close on a PR that a human can merge with one click.

### E7 — Close
Append to `theme-ledger.md` (decisions, deferred findings, next hints; ≤150 lines, summarize oldest away). **Tracker update — LOOP over EVERY member in `unit.yml` (a bundle closes ALL its tickets against the one held PR):** for each member ticket — LINEAR-LIVE → status In Review + a PR-URL comment (the same held PR # for every member); not-live → GitHub issue/PR body + one `tracker-debt.yml` row PER member (`{intended_op, linear_team, linear_project, title, github_artifact_url, evidence_urls, at, reconciled_at:null}`). Return to N4.

## 8. Stopping rules (checked N0 top + N4 close, priority order)
1. Kill switch → EXIT. 2. Queue empty → EXIT "ready for morning". 3. No-progress (night `no_progress_count ≥ 2`) → EXIT. 4. Per-unit fix caps (3/gate, 5/global; ≤2 CR fix rounds; 3-strike verification divergence) → HALT unit. 5. `[LAYER: checkpoint-resume]`: whole-iteration wall-clock **3h** → HALT unit. 6. Night wall-clock 8h → EXIT. 7. 10M-token night backstop → EXIT. 8. Blocked-by-human (hard-stop item needed mid-flight) → write `state/<run-id>/<ticket-id>/blocked.md`, HALT the unit, move on.

## 9. Failure protocol
HALT-PHASE = stop phase, append HALT to `progress.yml` + `night-log.md`, decrement fix budget, retry per cap. HALT-EXIT = write status naming what's blocked + why, EXIT — **do NOT tear down the sentinel first**; the Stop-hook audit needs it armed. 3+ failed fixes on one gate → STOP and question architecture. Never burn the night on one unit — a halted unit is skipped, its branch left for morning.
